# 레슨런 (Lessons Learned)

프로젝트 진행 중 배운 교훈, 반복하지 말아야 할 패턴, 더 나은 작업 방식을 기록한다.

---

## [2026-04-17] VM 배포로만 디버깅하다 커밋 20여 개 누적

### 상황

어제~오늘 사이 `fix:` 커밋 20건 이상 쏟아짐. 대부분 "VM 배포 → 실패 → 원인 추적 → 한 줄 수정 → 다시 배포" 루프를 반복한 결과.

### 원인

버그가 재현되는 환경을 잘못 이해하고 있었음.

- **오해**: "실제로 돌아가는 건 Oracle Cloud VM이니까 VM에서 테스트해야 한다."
- **실제**: 지금까지 고친 버그는 전부 **연수원 사이트(yeonsu.eseoul.go.kr)와 상호작용하는 로직 버그**였음 — 달력 판정, `change` 이벤트, 객실 선택 타이밍, Slack 포맷 등.
- 이 버그들은 로컬에서 돌리든 VM에서 돌리든 **같은 외부 사이트를 접속**하므로 **똑같이 재현**됨.

```
[로컬 PC]                    [Oracle VM]
    ↓ Playwright                ↓ Playwright
    └────→ yeonsu.eseoul.go.kr ←────┘
           (같은 DOM, 같은 JS 동작)
```

### 교훈

**"어디서 실행되는가"와 "버그를 어디서 재현할 수 있는가"는 다른 문제다.**

VM 환경 고유 버그(ARM64 Chromium, Docker 샌드박스, 네트워크)는 극소수. 대부분의 버그는 로컬에서 동일하게 재현되고, 오히려 **로컬 headful 브라우저로 눈으로 보면서 디버깅**하면 한 번에 여러 문제를 파악할 수 있다.

### 다음부터 이렇게

1. **로컬 headful 모드 먼저** — `HEADLESS=false uv run python main.py`로 브라우저 창 띄워 실제 DOM/JS 동작을 **눈으로 확인**. 한 번 실행에 여러 증거 수집.
2. **단계별 디버그 스크립트** — 예약 플로우 7단계 중 실패 단계만 독립 실행 (전체 플로우 반복 불필요).
3. **로깅 강화 후 1회 실행** — 실패 시 모든 단계의 DOM/URL/값을 덤프 → 다음 수정 시 여러 문제 동시 발견.
4. **VM 배포는 마지막 검증용 1회** — 로컬에서 동작 확인된 후에만.
